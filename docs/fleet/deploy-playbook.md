# Fleet Remote Deploy Playbook

Use this checklist to deploy a customer Fleet instance in one remote session and leave it running after disconnect.

## Scope

- Product command: `maestro-fleet deploy`
- Outcome:
  - Commander (`maestro-company`) configured
  - Initial project maestro optionally provisioned
  - Command Center reachable
  - OpenClaw gateway healthy
  - Detached Fleet web process running

See topology policy: `docs/fleet/supported-topologies.md`.

## Preconditions

Run these checks first on the customer machine:

```bash
python3 --version
node --version
npm --version
openclaw --version
tailscale ip -4
```

Required:
- Python 3.11+
- Node + npm
- OpenClaw CLI

Recommended:
- Tailscale connected (for tailnet URL handoff)

Critical topology rule:
- Same-user dual OpenClaw + Fleet is unsupported.
- If an existing shared OpenClaw gateway is active, stop and move to:
  - a fresh machine, or
  - a separate OS user account dedicated to Fleet.

## Remote Machine Bootstrap (First-Time Install)

Use this sequence after remoting into the customer server.

1. Export pinned release values (generated by `scripts/release-maestro-fleet.sh`):

```bash
export MAESTRO_FLEET_PACKAGE_SPEC="<root_wheel_url> <fleet_wheel_url>"
export MAESTRO_INSTALL_BASE_URL="<pinned_install_script_url>"
export MAESTRO_INSTALL_WRAPPER_URL="<pinned_linux_wrapper_script_url>"
```

2. Run the installer one-liner (no repo checkout on customer machine):

```bash
MAESTRO_FLEET_PACKAGE_SPEC="$MAESTRO_FLEET_PACKAGE_SPEC" \
MAESTRO_INSTALL_BASE_URL="$MAESTRO_INSTALL_BASE_URL" \
MAESTRO_FLEET_REQUIRE_TAILSCALE=1 \
curl -fsSL "$MAESTRO_INSTALL_WRAPPER_URL" | bash
```

The installer:
- verifies python/node/npm/openclaw (and tailscale if required)
- creates `~/.maestro/venv-maestro-fleet`
- installs pinned Fleet wheels
- launches `maestro-fleet deploy`

3. Verify binaries are available (if deploy was skipped):

```bash
~/.maestro/venv-maestro-fleet/bin/maestro-fleet --help
openclaw --version
```

### Fallback (Development Only)

Only use local repo checkout when testing development builds:

```bash
git clone <maestro-repo-url> ~/apps/maestro-fleet
cd ~/apps/maestro-fleet
MAESTRO_USE_LOCAL_REPO=1 bash scripts/install-maestro-fleet.sh
```

## Session Inputs to Collect

- Company name
- Commander model + default project model
- `GEMINI_API_KEY` (required for ingest/vision)
- `OPENAI_API_KEY` (recommended)
- `ANTHROPIC_API_KEY` (recommended; best commander instruction following)
- Commander Telegram bot token
- Initial project name + assignee
- Initial project Telegram bot token

## Primary Command (Interactive)

```bash
maestro-fleet deploy
```

## Primary Command (Non-interactive)

```bash
maestro-fleet deploy \
  --company-name "ACME Construction" \
  --commander-model "anthropic/claude-opus-4-6" \
  --project-model "openai/gpt-5.2" \
  --gemini-api-key "AIza..." \
  --openai-api-key "sk-..." \
  --anthropic-api-key "sk-ant-..." \
  --telegram-token "<company_bot_token>" \
  --project-name "Terminal Buildout" \
  --assignee "Sean" \
  --project-telegram-token "<project_bot_token>" \
  --non-interactive
```

## Offline / Local License Mode

Use this when deployment must avoid remote validation/payment checks:

```bash
maestro-fleet deploy ... --local
```

`--local` (alias `--offline`) enables local Fleet key generation/activation for project provisioning.

## OpenClaw Override Protection

If the machine appears to be managed by a non-Maestro OpenClaw default agent, deploy will fail closed.

Only override intentionally:

```bash
maestro-fleet deploy ... --allow-openclaw-override
```

## Verification Checklist

1. Profile and Fleet state:

```bash
maestro-fleet status
```

2. Doctor health:

```bash
maestro-fleet doctor --json
```

3. Command Center API:

```bash
curl -fsS http://localhost:3000/api/command-center/state | jq '.projects | length'
```

4. Detached process record:

```bash
cat ~/.maestro/fleet/serve.pid.json
tail -n 80 ~/.maestro/fleet/serve.log
```

5. Gateway status:

```bash
openclaw --profile maestro-fleet gateway status --json
```

## Customer Handoff Notes

Provide these in writing:

- Command Center local URL: `http://localhost:3000/command-center`
- Command Center tailnet URL (if connected): `http://<tailscale-ip>:3000/command-center`
- Commander Telegram handle: `@<commander_bot>`
- Initial project bot handle: `@<project_bot>`
- Log path: `~/.maestro/fleet/serve.log`
- PID state: `~/.maestro/fleet/serve.pid.json`

## Recovery Commands

Re-run doctor repair:

```bash
maestro-fleet doctor --fix
```

Re-run deployment (idempotent for existing setup):

```bash
maestro-fleet deploy
```

Restart background web server + re-verify:

```bash
maestro-fleet deploy
maestro-fleet doctor --json
```
